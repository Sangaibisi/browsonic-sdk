## 1. Protokol

### 1.1. Ne zaman yeni satır eklenir?

Her sprint kapanışında, [SPRINT_PLAN.md](./SPRINT_PLAN.md) Bölüm 1.3 (post-flight) gereği **mecburen**:

1. Sprint sırasında `browsonic-sdk` dışındaki bir repo'ya etki yarattıysa → aşağıdaki **Etki Tablosu**'na bir satır.
2. Etki yoksa → ilgili sprintin kapanış log'una "Bu sprintin cross-repo etkisi yok" yazılır, bu dosyaya satır eklenmez.

### 1.2. Ne zaman bir satır güncellenir?

İşletme aşamalarında (örn. service tarafı PR açıldı, ops tarafı CI'a entegre edildi):

- Status: `pending` → `in-progress` → `done`
- "İşletme Kaydı" kolonuna PR linki / commit hash eklenir
- Status `done` olduğunda satır tabloda kalır (geçmiş kayıt), silinmez.

## 3. Tarihsel Notlar

_Sprint kapanışlarında öğrenilen, ileride başka sprint'leri etkileyecek genel bulgular buraya yazılır. Tek-defa-not / lesson-learned formatı._

- **2026-04-29 — S1 kapanışı:** Cross-repo etki yok. SDK iç hijyen sprintiydi (license header + SPDX + ROADMAP + sprint tracking dosyaları). API yüzeyi, `/v1/events` ingest contract, ve build çıktıları değişmedi; service/dashboard/ops tarafında tetikleyici işletme yok. **Lesson-learned:** "saf iç temizlik" sprintleri için bu dosyaya satır eklemeden, SPRINT_PLAN.md'nin sprint kapanış log'una not düşmek protokol-uyumlu. Tablonun amacı sadece **gerçek aksiyon gerektiren** etkileri toplamak.
- **2026-04-29 — S5 kapanışı:** Yeni repo (Sangaibisi/browsonic-react) açıldı, 0.1 surface yayına hazır (boundary + 3 hook + HOC + demo + ADAPTER_TEMPLATE). **Lesson-learned-1:** Yeni paket açma sprintleri için `NPM_TOKEN` secret'ı CI'da bekleyen **gerçek bir aksiyon kaydı** üretiyor — entry #3 buna örnek. Gelecekteki adapter sprintleri (S6/S7/S10) aynı checklist'i tekrar açacak. **Lesson-learned-2:** `examples/**` dizini adapter'ın type-aware lint graph'ından hariç tutuldu — bu pattern'i ADAPTER_TEMPLATE.md'ye §6.1 olarak yansıttık, sonraki adapter'larda lint conflict yaşanmaması için.
- **2026-05-04 — S5.5 monorepo migration kapanışı:** `Sangaibisi/browsonic-sdk` reposu npm workspaces monorepo'ya çevrildi. SDK → `packages/sdk/`, React adapter → `packages/react/`. Eski standalone `Sangaibisi/browsonic-react` reposu önce `gh repo archive` ile arşivlendi, ardından kullanıcı tercihiyle `gh repo delete` ile **silindi** (lokal `browsonic-react/` dizini de `rm -rf` ile silindi, ~240M alan kazanıldı). **Lesson-learned-1:** S5'te öğrenilen "her adapter için yeni repo + NPM_TOKEN + secret + CI duplicate" yükü, monorepo'da tek-token tek-CI ile ortadan kalkıyor. S5.5 timing kritik: 7 adapter açılmadan **erken** geçilmesi 7×iş katsayısını engelledi. **Lesson-learned-2:** Cross-repo `git mv` mümkün değil, dolayısıyla adapter'ın per-file commit history'si monorepo'ya taşınamaz. Default tavsiye: **archive** — eski commit hash referansları (CHANGELOG, GitHub Release, npm provenance, dış blog/docs linkleri) ancak archive'da erişilebilir kalıyor. **İstisna**: hiç npm publish edilmemiş + GitHub Release oluşmamış (`latestRelease: null`) + dış link havuzu yok ise `gh repo delete` kabul edilebilir tercih — tek cost SPRINT_PLAN.md'deki commit hash referanslarının link rot olması (metin tarihsel kayıt için kalır). browsonic-react bu istisnaya uydu. **Lesson-learned-3:** ADAPTER_TEMPLATE'in §0 + §1 + §8 + §9 bölümleri monorepo akışına yeniden yazıldı; gelecek S6/S7/S10 sprintleri "yeni repo aç" yerine "`mkdir packages/<framework>`" ile başlıyor — checklist 1-2 saatten 5 dakikaya iniyor.
- **2026-05-04 — S8 kapanışı:** Public scope/breadcrumb/tag API tamamen yayında — `setTag` / `setContext` / `setExtra` (M1, `cd7f53f`), `addBreadcrumb` (M2, `2f2517a`), `withScope` sync+async (M3). Cross-repo etkisi **additive ingest tolerance** — service tarafı 3 yeni alanı (`contexts`, `extras`, `telemetry.breadcrumb`) absorb etmek yeterli; persist için 2 yeni JSONB sütunu + telemetry kolonunda dolaylı yapı genişlemesi (entry #6/7). Dashboard tarafı opsiyonel UI (entry #9). **Lesson-learned-1:** "Sentry parite" sprintleri (yüzey eklemek), bundle bütçesini en zor yoran iş tipi değil — toplam +0.62 KB main, +0.57 KB core (M1+M2+M3), bütçenin 1/4'ü kullanıldı. Eski sürümlerde her API metodunun ~1KB civarı maliyet hipotezi gerçekçi değildi; tree-shake + interface-only types için cost ~0.1–0.25 KB. **Lesson-learned-2:** `withScope` async overload tasarımı, return değerinin runtime'da thenable kontrolü ile dallanması yaklaşımı, derleyici-tarafında iki overload signature ile ergonomic ve eski tarayıcılarda Promise.finally polyfill bağımlılığı sıfır. Gelecek transient-state API'leri (örn. `withTransport`, `withSession`) bu kalıbı tekrar kullanabilir. **Lesson-learned-3:** Per-feature snapshot + restore pattern, "scope" abstraction'ı için 25 satır kod yetiyor — Sentry'nin Scope class hierarchy'si bizim kullanım hacmimize aşırı ağır. Genişlerse refactor maliyeti düşük. **Lesson-learned-4:** Breadcrumb wire format'ı, mevcut `TelemetryTimeline`'ın 5. kanal genişlemesi olarak yapıldı — yeni event type yaratmak yerine var olan timeline yapısına ekleme yapmak, dashboard renderer ve service ingest tarafında farkındalık eşiğini düşürdü. Bu pattern (mevcut struct'a kanal eklemek vs yeni struct yaratmak) S9'daki session health verisi için de geçerli olabilir.
- **2026-05-04 — S6 kapanışı:** İki adapter — `@browsonic/vue` (M1, `b83c1bf`) + `@browsonic/svelte` (M2, `3c2a9df`) — `packages/vue/` ve `packages/svelte/` workspaces olarak yayında. Cross-repo etkisi minimum: service ve dashboard tarafı **dokunmadı** (adapter'lar SDK API'sini tüketiyor; wire format değişmedi — entry #11 izleme satırı). Landing tarafı opsiyonel CTA güncelleme (entry #10). **Lesson-learned-1:** ADAPTER_TEMPLATE.md'nin S5.5'te güncellenmesi (`mkdir packages/<framework>`) gerçekten checklist süresini düşürdü — Vue M1 ve Svelte M2 her biri ~30 dakika bootstrap + ~30 dakika src + ~30 dakika test = ~1.5 saat/adapter. Tek-repo + tek-NPM_TOKEN modeli ekstra setup'ı sıfıra indirdi. **Lesson-learned-2:** Vue test setup'ında `happy-dom` window'unun afterEach'te `delete window` ile yıkılması diğer test dosyalarını kırdı (vitest happy-dom global'ı per-file kuruyor, per-test değil). Doğru pattern: `delete (window as ...).Browsonic` — sadece custom property'yi sil, window kendisini bırak. **Lesson-learned-3:** Vue `errorCaptured` + setup composition reactive update (re-render) bir sonraki tick'te oluyor — boundary fallback testi için `await nextTick()` gerekiyor. Bu detayı ADAPTER_TEMPLATE.md §4'e dipnot olarak eklenebilir (S7 hazırlığında değerlendir). **Lesson-learned-4:** Svelte adapter'ı için boundary component shipped EDİLMEDİ — Svelte 5'in native `<svelte:boundary>` primitive'i framework feature olarak yetiyor; Svelte 4'te temiz alternatif yok. AGENTS.md'nin "Divergences from ADAPTER_TEMPLATE" bölümü bu kararı dokümante ediyor. **Bu örnek**, ADAPTER_TEMPLATE'in bir contract değil checklist olduğunu gösteriyor — framework idiomları zorladığında bilinçli divergence kabul edilir, yeter ki gerekçesi yazılsın. **Lesson-learned-5:** Pure-TS adapter (Svelte gibi) build chain'i `tsc` × 3'te kalır → consumer tarafında bundler-side compiler bağımlılığı sıfır. `.svelte` / `.vue` SFC dosyaları shipping etmek için consumer bundler'ları compile-time desteğe ihtiyaç duyuyor — adapter'ı küçük tutmak için pure-TS render fonksiyonları tercih et. Sadece SFC ergonomi gerçekten gerektiğinde (örn. CSS scoping) compiler'ı build chain'e ekle.
- **2026-05-04 — S10 kapanışı + 19-week plan complete:** İki adapter — `@browsonic/angular` (M1, `ff106f7`) + `@browsonic/remix` (M2, `c134b89`) — ve iki migration guide (`MIGRATION_FROM_SENTRY.md` + `MIGRATION_FROM_TRACKJS.md`) yayında. **Browsonic framework matrix'i tamamlandı** — 7 adapter (react / vue / svelte / nextjs / astro / angular / remix) + SDK core. Cross-repo etkisi yok (entry #17/#18 dogfooding fırsatı, opsiyonel). **Lesson-learned-1:** 19-haftalık planın **fiili tamamlanma süresi: ~3 saat session work** (S1 başlangıç → S10 kapanış, takvim haftaları yerine işin gerçekten ne kadar sürdüğü). Pattern: monorepo (S5.5'te kuruldu) + ADAPTER_TEMPLATE.md (S5'te yazıldı, S5.5'te genişletildi) + Conventional Commits + semantic-release + size-limit + per-package vitest disiplini. Bu altyapı her yeni adapter'ı 30-90 dakikaya indiriyor — önceki Sentry/Sentry-style kurulumda tek adapter ~bir gün çekecekti. **Lesson-learned-2:** Pure-TS adapter pattern'i (Svelte, Astro, Angular) `.<framework>` SFC dosyaları shipping etmekten kesin daha hafif: tek build chain (`tsc` × 3), no compiler dep, daha küçük bundle. Sadece SFC ergonomi GERÇEKTEN gerekiyorsa (örn. Vue SFC + CSS scoping default fallback) compiler eklenir; biz tüm adapter'larda render functions ile kaçındık. **Lesson-learned-3:** Meta-framework adapter pattern (`@browsonic/nextjs → @browsonic/react`, `@browsonic/remix → @browsonic/react`) hem DX (consumer install one package) hem maintenance (peer-dep range tek lokasyonda) açısından kazandı. Sentry pattern'i doğrulandı. Astro standalone (multi-framework) ve Angular standalone (kendi DI) farklı modeller — adapter design choice'u host framework architecture'a göre. **Lesson-learned-4:** Migration guide'ları kısa tuttuk (her biri ~10 step + mapping tablosu) — codemod scripti deferred. Mapping tabloları kapsamlı yazılınca manual migration ~1 saat içinde yapılabiliyor; codemod opsiyonel future feature. **Lesson-learned-5:** Per-framework quickstart dosyaları (`docs/quickstart/<framework>.md`) yazılmadı çünkü her paket README'sinde **zaten** Quickstart bölümü var — duplicate olur. Migration README'sinden direkt paket README'lerine link verildi. Documentation duplikasyonundan kaçınmak duraklayıp düşünmeyi gerektirdi; "her sprint'in plan kalemi YAZILMASI ŞART değildir" prensibi.
- **2026-05-04 — S9 partial closure (M1+M2 KAPANDI, M3 ERTELENDİ):** SDK içinde iki feature shipped — runtime environment guards (extension + bot detection at init time) ve session health state machine. Cross-repo etkisi: service tarafı `BrowsonicEvent.sessionHealth` opsiyonel alanı tolerate etmeli (entry #14); dashboard opsiyonel UI (entry #15); ops M3 reopen blocker'ı (entry #16, CDN distribution kurulduğunda). **Lesson-learned-1:** Bir sprint'in tüm milestone'ları tamamlanmak zorunda değil — M3 (CDN loader script) ops bağımlılığı (CDN distribution kanalı) kurulmadan tamamlanamaz, dolayısıyla "ERTELENDİ" + reopen kriteri docs'a yazıldı. Kabul edilebilir bir kalıp; S3+S4 (source map pipeline) zaten benzer şekilde ERTELENDİ ile bekliyor (orada design review bekliyordu, burada infra dependency). **Lesson-learned-2:** Init-time guard'lar, event-time filter'larından FARKLI bir amaca hizmet ediyor — `ignoreExtensions` (event-time) extension origin'li event'leri capture edip backend'e gitmeden filter ediyor; `abortInExtensionContext` (init-time) extension context'te tüm SDK pipeline'ını açmıyor. İkisi tamamlayıcı, biri diğerinin yerine geçmiyor. Bu ayrım API doc'larında net yapıldı. **Lesson-learned-3:** Default 28 bot pattern, conservative scope ile başlandı (major search engines + social link previews + headless tooling). Aggressive filtering isteyen kullanıcılar `botPatterns` ile composing yapıyor — `[...DEFAULT_BOT_PATTERNS, 'mybot']`. False-positive raporları gelirse default listeyi daraltırız. **Lesson-learned-4:** Session health state machine'i **monotonic terminal-`'crashed'`** ile tasarlandı — bir kez crashed olunca recovery yok. Bu Sentry'nin "release health" özelliğindeki "crashed session" tanımıyla uyumlu (oradaki tanım da: bir session içinde herhangi bir error → errored, SDK self-failure / unrecoverable → crashed). Numeric `severity()` aggregation backend tarafında comparator olarak yararlı.
- **2026-05-04 — S7 kapanışı:** İki meta-framework adapter — `@browsonic/nextjs` (M1, `db213f9`) + `@browsonic/astro` (M2, `0ebfeaa`) — `packages/nextjs/` ve `packages/astro/` workspaces olarak yayında. Cross-repo etkisi yok: wire format değişmedi (entry #12 + #13 dogfooding fırsatları, opsiyonel). **Lesson-learned-1:** Meta-framework adapter pattern: Next.js apps React, Astro multi-framework. Next adapter `@browsonic/react`'ı **peer + dev** olarak alıp tüm yüzeyi re-export ediyor (Sentry'nin `@sentry/nextjs → @sentry/react` ilişkisi). Astro tek bir framework'e bağlanmadığı için kendi yüzeyini ship ediyor (View Transitions + capture wrappers); per-framework boundary kullanıcının import ettiği framework adapter'ında. Bu iki kalıp S10'da (Angular + Remix) tekrar kullanılacak — Remix React adapter'a depend ediyor, Angular standalone. **Lesson-learned-2:** Naming collision — React adapter `withBrowsonic` HOC sunuyor; Next config wrapper aynı ismi alamaz. Sentry pattern (`withSentryConfig`) ile çözüldü: `withBrowsonicConfig`. Re-export'lar HOC'u orijinal isminde geçirdi. Gelecek meta-framework'ler için aynı naming dispatch (`with<Framework>Config`) kullanıla. **Lesson-learned-3:** Astro test'inde `document.addEventListener` listener'larını test'ler arası temizlemek için per-test `subscriptions: Array<() => void>` + `afterEach` cleanup pattern eklendi. happy-dom listener'ları reset etmiyor; explicit `track(off)` discipline gerekiyor. Bu pattern S10'daki Remix adapter'ı için de geçerli (Remix client navigation hook listener'ları aynı sorunu yaşayacak). **Lesson-learned-4:** ESLint `@typescript-eslint/require-await` kuralı, async-throw test handler'larında (`async () => { throw err }`) tetikleniyor. Test files için bu kuralı off ettik — test ergonomi için yararlı. Production code'da kural aktif kalıyor. **Lesson-learned-5:** Re-export pattern bundle bütçesini tüketmiyor — `@browsonic/nextjs`'in dist'i sadece kendi src'sini içeriyor, `@browsonic/react` consumer install zamanında ayrı resolve oluyor (workspace symlink dev'de, npm dependency prod'da). 6 adapter (react/vue/svelte/nextjs/astro + s10'da angular+remix) yan yana yaşarken kullanıcı sadece kendi seçtiğini install ediyor.

## 4. Şablon (Yeni Satır Eklerken)

Yeni bir etki kaydı eklemek için Bölüm 2'deki tabloya aşağıdaki şablonu kullanarak bir satır ekle:

```
| <sıra> | S<sprint#> | YYYY-MM-DD | <repo-adı> | <iş tek cümle> | pending | <kim/hangi-repo> | <PR linki veya boş> |
```

Örnek (gerçek değil, sadece format):

```
| 1 | S3 | 2026-05-12 | browsonic-service | POST /v1/sourcemaps endpoint'ini implemente et (multipart form, debugId, checksum) | pending | service-team | — |
| 2 | S3 | 2026-05-12 | browsonic-ops | CI release flow'una `browsonic-sourcemaps upload` adımını ekle | pending | ops-team | — |
```

Status değişiminde aynı satırı güncelle, **yeni satır açma**.
